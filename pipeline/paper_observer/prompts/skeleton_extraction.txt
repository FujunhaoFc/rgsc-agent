# Field Guidance

- problem_definition.task: ONE sentence stating what task the paper addresses.
- problem_definition.input / output: what goes in and comes out of the proposed method.
- problem_definition.evaluation_target: what the paper aims to demonstrate (e.g. "match retrain-from-scratch behavior in machine unlearning").

- core_method.name: short identifier (e.g. "AMUN", "min-p sampling").
- core_method.one_line: a single sentence summarizing what the method does mechanically.
- core_method.settings[]: list of distinct configurations/variants tested. ONE entry per setting (e.g. "with remain set", "without remain set"). If the paper has only one setting, return a single entry. If none, return [].
- core_method.key_hyperparameters: 3-8 most important hyperparameters with concrete values from the paper.

- datasets: each dataset USED in experiments (not just mentioned in related work). "used_for" describes the role (e.g. "main classification benchmark", "validation").

- models: backbone models used in experiments (e.g. "ResNet-18", "Mistral 7B"). "role" is the model's role (e.g. "classifier", "language model").

- baselines: methods compared against the proposed method (NOT the proposed method itself). Include ALL methods that appear in the comparison tables/figures, including:
  - Approximate methods (e.g. FT, RL, GA, BS, SalUn for unlearning; or comparable named baselines in other domains).
  - Exact / oracle methods (e.g. "Retrain", "Retrain-from-scratch") — even if commonly treated as an upper-bound oracle, if the paper compares against it numerically, INCLUDE it as a baseline.
  - Any prior method explicitly named as a comparison point.
  Do NOT include the proposed method itself in this list.
  "type" is a short categorization (e.g. "exact unlearning", "approximate unlearning", "weight saliency", "nucleus sampling").

- metrics: evaluation metrics used. "definition" is a brief explanation of what the metric measures.

- main_experiments: ONE entry per Table or Figure that presents experimental results, including:
  - All tables/figures showing numerical comparisons against baselines.
  - All tables/figures explicitly cited in the abstract, intro, or conclusions as evidence for the paper's claims.
  - Tables presenting "main results", "ablation results", and "no-access" or "without X" condition results — these all count.
  - Figures whose subplots show numerical or empirical claims that the paper relies on.
  Do NOT include purely qualitative diagrams (architecture diagrams, schematic illustrations, t-SNE plots without numerical claims).
  Cross-check yourself: every "execution outcomes match Table N" or "match Figure N" claim that the paper would make MUST have a corresponding main_experiments entry.
  Use "primary": true for results that support the central claim; "primary": false for ablations and supplementary results.
  Aim for ~5-10 entries for a typical ML paper.
   - "evidence_in_paper" must match an entity id from ENTITIES.
   - "claim" is a brief description of what this experiment shows.
   - "settings_used", "datasets_used", "models_used" are arrays of names matching settings/datasets/models defined above.
   - "axes" MUST be null in this Call.

- expected_claims: 3-8 specific factual claims the paper makes that would be verified by reproducing the experiments. Each claim should be checkable against actual numerical/qualitative results.
   - "evidence" must be an entity id from ENTITIES.
   - "verification_hint" is a short instruction on how to check the claim from results data.

- reproducibility_meta:
   - "needs_training": true if any model training/fine-tuning is required to reproduce.
   - "estimated_compute": short description (e.g. "single GPU, hours", "8 GPUs, days", "CPU only, minutes").
   - "data_availability": MUST be one of "public", "partial", "private", "unknown".
   - "code_partial_available": true if the paper mentions an official code release.
   - "minimum_viable_experiments": list of main_experiment ids that MUST be reproduced to validate the central claim. Should include all primary=true experiments that test the main proposed method on the main dataset.
   - "deferred_experiments": main_experiment ids that can be skipped under tight compute budget — typically ablations on extra datasets, model variants, or supplementary analyses. Do NOT defer experiments that are part of the main claim or that the paper highlights as supporting central conclusions.
